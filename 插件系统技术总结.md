## 1. 整体架构概览

```
┌─────────────────────────────────────────────────────────────┐
│                        MCP Server                           │
│                                                             │
│  main.cpp                                                   │
│    ├─ PluginsLoader  ──→  扫描 plugins/ 目录                │
│    │     ├─ dlopen() 加载 .so 文件                           │
│    │     ├─ dlsym()  获取 CreatePlugin / DestroyPlugin       │
│    │     └─ CreatePlugin() 创建 PluginAPI* 实例              │
│    │                                                         │
│    └─ Server  ──→  接收请求 → GetPluginsSnapshot()           │
│                       → 遍历插件 → HandleRequest()           │
└─────────────────────────────────────────────────────────────┘

plugins/
├── weather/libweather.so      ← 共享库插件
├── sleep/libsleep.so
├── notification/libnotification.so
├── code-review/libcodereview.so
└── .staging/                  ← 运行时临时副本（自动管理）
```

**核心设计思想**：主程序通过**纯 C 接口**与插件通信，插件编译为**共享库（.so）**，运行时通过 `dlopen`/`dlsym` 动态加载，实现主程序与插件的**二进制解耦**。

## 2. PluginAPI.h —— 插件接口定义

### 2.1 文件位置

```
src/interface/PluginAPI.h
```

### 2.2 完整结构解析

```c
// 符号可见性控制：标记哪些符号需要从 .so 中导出
#define PLUGIN_API __attribute__((visibility("default")))

// 通知回调函数类型
typedef void (*ClientNotificationCallback)(const char* pluginName, const char* notification);

// 插件类型枚举：三种 MCP 能力
typedef enum {
    PLUGIN_TYPE_TOOLS     = 0,   // 工具调用
    PLUGIN_TYPE_PROMPTS   = 1,   // 提示模板
    PLUGIN_TYPE_RESOURCES = 2    // 资源访问
} PluginType;

// 工具描述结构体
typedef struct {
    const char* name;           // 工具名称，如 "get_weather"
    const char* description;    // 工具描述
    const char* inputSchema;    // JSON Schema 格式的参数描述
} PluginTool;

// 提示描述结构体
typedef struct {
    const char* name;
    const char* description;
    const char* arguments;      // JSON 格式的参数列表
} PluginPrompt;

// 资源描述结构体
typedef struct {
    const char* name;
    const char* description;
    const char* uri;            // 资源 URI
    const char* mime;           // MIME 类型
} PluginResource;

// 通知系统（宿主注入给插件的回调）
typedef struct {
    ClientNotificationCallback SendToClient;
} NotificationSystem;

// ============ 核心：插件接口表（虚函数表的 C 风格实现）============
typedef struct {
    // --- 元数据 ---
    const char* (*GetName)();           // 插件名称
    const char* (*GetVersion)();        // 插件版本
    PluginType  (*GetType)();           // 插件类型

    // --- 生命周期 ---
    int         (*Initialize)();        // 初始化，返回 1 成功
    char*       (*HandleRequest)(const char* request);  // 处理请求，返回 JSON 字符串
    void        (*Shutdown)();          // 关闭清理

    // --- 能力声明 ---
    int              (*GetToolCount)();            // 工具数量
    const PluginTool* (*GetTool)(int index);       // 获取第 i 个工具
    int              (*GetPromptCount)();          // 提示数量
    const PluginPrompt* (*GetPrompt)(int index);   // 获取第 i 个提示
    int              (*GetResourceCount)();        // 资源数量
    const PluginResource* (*GetResource)(int index); // 获取第 i 个资源

    // --- 宿主注入 ---
    NotificationSystem* notifications;  // 宿主设置的通知回调
} PluginAPI;

// ============ 导出函数（每个 .so 必须提供）============
PLUGIN_API PluginAPI* CreatePlugin();   // 创建插件实例
PLUGIN_API void DestroyPlugin(PluginAPI*); // 销毁插件实例
```

### 2.3 关键设计分析

**为什么用 C 接口而不用 C++ 虚函数？**

| 特性 | C 接口（当前方案） | C++ 虚函数 |
|------|-------------------|-----------|
| ABI 稳定性 | ✅ C ABI 跨编译器稳定 | ❌ C++ ABI 各编译器不同 |
| 符号名 | ✅ `extern "C"` 保持原始名 | ❌ 名称修饰(Name Mangling) |
| 跨语言 | ✅ Python/Rust 等可调用 | ❌ 仅限 C++ |
| 版本兼容 | ✅ 函数指针表可扩展 | ❌ 虚表布局变化会崩溃 |

**函数指针表**本质上是**手动虚表（vtable）**，每个插件填充自己的函数指针，主程序通过统一接口调用——这是插件系统最常见的设计模式。

### 2.4 `__attribute__((visibility("default")))` 详解

```c
#define PLUGIN_API __attribute__((visibility("default")))
```

- **作用**：标记符号为**可见（导出）**，使其可被 `dlsym()` 找到
- **背景**：编译时加 `-fvisibility=hidden` 默认隐藏所有符号，只有标记 `visibility("default")` 的符号才导出
- **好处**：
  - 减少符号表大小，加快加载速度
  - 避免符号冲突（同名函数在不同 .so 中不会互相干扰）
  - 缩小 .so 文件体积

## 3. CMake 构建系统：动态库与静态库

### 3.1 项目整体 CMake 结构

**根 CMakeLists.txt**（主程序）：

```cmake
cmake_minimum_required(VERSION 3.20.0)
project(mcp_server)

set(CMAKE_CXX_STANDARD 20)

# 主程序源文件
set(SOURCES
    src/main.cpp
    src/server/Server.cpp
    src/loader/PluginsLoader.cpp
    # ... 其他源文件
)

# 主程序构建为可执行文件
add_executable(${PROJECT_NAME} ${SOURCES})

# 包含头文件搜索路径
target_include_directories(${PROJECT_NAME} PRIVATE
    include
    src
    src/interface
    src/utils
    libs_tier_01/aixlog-1.5.0/include
    ${CMAKE_CURRENT_BINARY_DIR}
)

# 链接 pthread
target_link_libraries(${PROJECT_NAME} PRIVATE Threads::Threads)

# Linux 下静态链接 libgcc 和 libstdc++
if(CMAKE_COMPILER_IS_GNUCC OR CMAKE_CXX_COMPILER_ID MATCHES "Clang")
    if(UNIX AND NOT APPLE)
        target_link_options(${PROJECT_NAME} PRIVATE "-static-libgcc" "-static-libstdc++")
    endif()
endif()

# 引入各插件子目录
add_subdirectory(plugins/sleep)
add_subdirectory(plugins/notification)
add_subdirectory(plugins/weather)
add_subdirectory(plugins/code-review)
```

### 3.2 插件 CMakeLists.txt（以 weather 为例）

```cmake
cmake_minimum_required(VERSION 3.10)

# ========== 动态库（Shared Library）==========
add_library(weather SHARED
    ${PROJECT_SOURCE_DIR}/plugins/weather/Weather.cpp    
)

# 关键：位置无关代码，.so 必须
set_target_properties(weather PROPERTIES POSITION_INDEPENDENT_CODE ON)

# 链接依赖
find_package(Threads REQUIRED)
target_link_libraries(weather PRIVATE Threads::Threads)

# 可选：预处理器宏定义
target_compile_definitions(weather PRIVATE WEATHER_EXPORTS)

# 包含插件接口头文件
target_include_directories(weather PRIVATE 
    ${PROJECT_SOURCE_DIR}/include 
    ${PROJECT_SOURCE_DIR}/src/interface
)
```

### 3.3 `add_library` 的 SHARED vs STATIC

```cmake
# 动态库（运行时加载，.so / .dll / .dylib）
add_library(weather SHARED Weather.cpp)

# 静态库（编译时链接，.a / .lib）
add_library(weather STATIC Weather.cpp)
```

**对比**：

| 特性 | SHARED（动态库） | STATIC（静态库） |
|------|-----------------|-----------------|
| 文件扩展名 | `.so`（Linux）/ `.dll`（Windows）/ `.dylib`（macOS） | `.a`（Linux）/ `.lib`（Windows） |
| 链接时机 | 运行时（dlopen）或加载时 | 编译时 |
| 文件大小 | 较小，共享代码段 | 较大，代码被复制进可执行文件 |
| 更新方式 | 替换 .so 文件即可 | 需要重新编译主程序 |
| 符号冲突 | 不同 .so 互相隔离 | 可能与其他库冲突 |
| 适用场景 | **插件系统**、共享库 | 库的发布、不需要动态加载 |
| 位置无关代码 | **必须** `-fPIC` | 不需要 |
| 部署 | 需要确保 .so 在 LD_LIBRARY_PATH 中 | 打进可执行文件，无额外依赖 |

### 3.4 `POSITION_INDEPENDENT_CODE` 的作用

```cmake
set_target_properties(weather PROPERTIES POSITION_INDEPENDENT_CODE ON)
```

- 等价于编译选项 `-fPIC`
- **为什么 .so 必须**：动态库加载到内存的地址在编译时不确定，PIC 代码通过 GOT（全局偏移表）和 PLT（过程链接表）间接寻址，使得代码可以在任意地址重定位
- **静态库**通常不需要（除非被动态库链接）

### 3.5 编译命令

```bash
# 创建构建目录
mkdir build && cd build

# 生成构建文件
cmake .. -DCMAKE_BUILD_TYPE=Release

# 编译
cmake --build . -j$(nproc)

# 产物
# 主程序: build/mcp_server
# 插件:   build/plugins/weather/libweather.so
#         build/plugins/sleep/libsleep.so
#         build/plugins/notification/libnotification.so
#         build/plugins/code-review/libcodereview.so
```

### 3.6 如果要构建静态库

修改插件的 CMakeLists.txt：

```cmake

# 方式二：通过选项控制（推荐）

    add_library(weather SHARED Weather.cpp)

    add_library(weather STATIC Weather.cpp)
endif()

```

## 5. 静态库如何暴露接口

### 5.1 核心问题

静态库编译时链接，**没有运行时符号发现机制**（不能 `dlsym`），所以需要一种方式让主程序"找到"插件。

### 5.2 方案一：手动注册（当前项目的静态化版本）

**PluginAPI.h 保持不变**，插件实现保持不变，但主程序需要主动调用：

```cpp
// main.cpp 中手动注册静态插件
extern "C" PluginAPI* CreatePlugin();  // 声明（静态库已提供实现）

int main() {
    // 直接调用，无需 dlopen
    PluginAPI* plugin = CreatePlugin();
    plugin->Initialize();
    // ... 使用插件
}
```

### 5.3 方案二：注册表模式（更优雅）

```cpp
// PluginRegistry.h
class PluginRegistry {
    static std::vector<std::function<PluginAPI*()>>& GetCreators() {
        static std::vector<std::function<PluginAPI*()>> creators;
        return creators;
    }
public:
    static void Register(std::function<PluginAPI*()> creator) {
        GetCreators().push_back(creator);
    }
    static const auto& GetCreators() { return GetCreators(); }
};

// 每个插件中的静态注册（在头文件中）
#define REGISTER_PLUGIN(FactoryFn) \
    static bool _registered = [] { \
        PluginRegistry::Register(FactoryFn); \
        return true; \
    }();

// Weather.cpp 中
REGISTER_PLUGIN([]() -> PluginAPI* { return CreatePlugin(); });

// main.cpp 中
for (auto& creator : PluginRegistry::GetCreators()) {
    PluginAPI* plugin = creator();
    plugin->Initialize();
}
```

### 5.4 方案三：通过链接器 section 自动注册

```c
// 利用 ELF section 在程序启动前自动收集所有插件
#define REGISTER_PLUGIN(func) \
    __attribute__((used, section("plugins"))) \
    static PluginAPI* (*_plugin_##func)() = func;

// 加载时遍历 section
extern PluginAPI* (*__start_plugins)();
extern PluginAPI* (*__stop_plugins)();

void load_all_plugins() {
    for (auto** p = &__start_plugins; p < &__stop_plugins; p++) {
        PluginAPI* plugin = (*p)();
        plugin->Initialize();
    }
}
```

### 5.5 动态库 vs 静态库的接口暴露对比

| | 动态库 (.so) | 静态库 (.a) |
|---|---|---|
| **符号导出** | `__attribute__((visibility("default")))` | 默认全部导出 |
| **接口发现** | `dlsym(handle, "CreatePlugin")` | 编译时直接调用 / 注册表 |
| **头文件** | 只需 PluginAPI.h | 需要 PluginAPI.h + 插件声明头文件 |
| **灵活性** | 运行时按需加载 | 编译时确定 |
| **部署复杂度** | 需要管理 .so 文件路径 | 打进一个可执行文件 |
| **更新** | 替换 .so 无需重编主程序 | 必须重新编译链接 |

## 6. Linux 动态加载机制：dlopen / dlsym / dlclose

### 6.1 API 详解

```c
#include <dlfcn.h>

// 打开动态库
void* dlopen(const char* filename, int flag);
// filename: .so 文件路径
// flag:     RTLD_LAZY（懒加载）| RTLD_NOW（立即加载）
// 返回:     库句柄（void*），失败返回 NULL

// 查找符号
void* dlsym(void* handle, const char* symbol);
// handle: dlopen 返回的句柄
// symbol: 函数或变量名（字符串）
// 返回:   符号地址，失败返回 NULL

// 关闭动态库
int dlclose(void* handle);
// handle: dlopen 返回的句柄
// 返回:   0 成功，非 0 失败

// 获取错误信息
const char* dlerror(void);
// 返回:   最近一次错误的描述字符串
```

### 6.2 本项目中的使用

```cpp
// PluginsLoader.cpp 中的关键代码

// 第一步：打开 .so 文件（懒加载模式）
entry->handle = dlopen(stagingPath.c_str(), RTLD_LAZY);

// 第二步：查找导出函数
entry->createFunc  = (PluginAPI* (*)())dlsym(entry->handle, "CreatePlugin");
entry->destroyFunc = (void (*)(PluginAPI*))dlsym(entry->handle, "DestroyPlugin");

// 第三步：调用工厂函数创建实例
entry->instance = entry->createFunc();

// 第四步：初始化
entry->instance->Initialize();

// 卸载时（PluginEntry 析构函数中）：
entry->instance->Shutdown();     // 1. 插件清理
destroyFunc(entry->instance);    // 2. 销毁实例
dlclose(handle);                 // 3. 关闭 .so
```

### 6.3 函数指针类型转换解析

```c
// CreatePlugin 的函数指针类型
PluginAPI* (*createFunc)()
//         ↑              ↑ 无参数
//         └ 返回 PluginAPI*

// 写成 typedef 更清晰：
typedef PluginAPI* (*CreatePluginFunc)();
typedef void (*DestroyPluginFunc)(PluginAPI*);

CreatePluginFunc createFunc = (CreatePluginFunc)dlsym(handle, "CreatePlugin");
DestroyPluginFunc destroyFunc = (DestroyPluginFunc)dlsym(handle, "DestroyPlugin");
```

### 6.4 dlopen 搜索路径规则

```bash
# 搜索顺序：
1. 如果路径是绝对路径或相对路径，直接使用该路径
2. LD_LIBRARY_PATH 环境变量
3. /etc/ld.so.cache（由 ldconfig 生成）
4. 默认路径：/lib, /usr/lib

# 常用方式：
dlopen("./plugins/libweather.so", RTLD_LAZY)   # 相对路径
dlopen("/usr/lib/mcp-plugins/libweather.so", RTLD_LAZY)  # 绝对路径
dlopen("libweather.so", RTLD_LAZY)              # 系统搜索

# Linux 特有：
dlopen("libweather.so", RTLD_LAZY | RTLD_DELEGATE_SYMBOLS)  # 级联查找符号
```

### 6.5 符号可见性编译选项

```bash
# 默认：所有符号可见（容易冲突）
g++ -shared -o libweather.so Weather.cpp

# 推荐：隐藏所有，只导出标记的
g++ -shared -fvisibility=hidden -o libweather.so Weather.cpp
# 此时只有 __attribute__((visibility("default"))) 标记的符号可被 dlsym 找到

# 查看 .so 导出的符号
nm -D libweather.so
# 或
readelf -Ws libweather.so | grep FUNC
```

## 7. 加载入口：CreatePlugin / DestroyPlugin

### 7.1 工厂函数模式

每个插件必须导出两个 C 函数：

```cpp
// 插件内部（Weather.cpp）

// 静态全局实例（注意：这决定了 staging 机制的必要性）
static PluginAPI plugin = {
    GetNameImpl,      GetVersionImpl,   GetTypeImpl,
    InitializeImpl,   HandleRequestImpl, ShutdownImpl,
    GetToolCountImpl, GetToolImpl,
    nullptr,          nullptr,           nullptr, nullptr
};

// 工厂函数：返回静态实例的指针
extern "C" PLUGIN_API PluginAPI* CreatePlugin() {
    return &plugin;
}

// 析构函数：在此处清理资源
extern "C" PLUGIN_API void DestroyPlugin(PluginAPI* p) {
    // 如果是 new 出来的，这里 delete
    // 当前方案返回静态对象，所以为空
}
```

### 7.2 为什么需要 staging 机制

```
问题场景：
  第一次 dlopen("plugins/libweather.so")  → 加载成功，得到 plugin 指针
  重新编译插件 → 覆盖了 libweather.so 文件
  第二次 dlopen("plugins/libweather.so")  → 操作系统返回已缓存的旧模块！
                                           不会加载新文件！

解决（staging 机制）：
  复制到 .staging/libweather_1719000000000.so  ← 唯一文件名
  dlopen(".staging/libweather_1719000000000.so") ← 新文件，新模块
  卸载时 dlclose + 删除副本
```

**原理**：Linux 的 `dlopen` 对**同一路径**有模块缓存，对**不同路径**创建新模块。用纳秒时间戳保证路径唯一。

### 7.3 完整的插件实现模板

```cpp
#include "PluginAPI.h"
#include "json.hpp"
using json = nlohmann::json;

// 1. 定义工具列表
static PluginTool tools[] = {
    {
        "my_tool",
        "Tool description here",
        R"({
            "$schema": "http://json-schema.org/draft-07/schema#",
            "type": "object",
            "properties": {
                "param1": {"type": "string"}
            },
            "required": ["param1"],
            "additionalProperties": false
        })"
    }
};

// 2. 实现接口函数
const char* GetNameImpl() { return "my-plugin"; }
const char* GetVersionImpl() { return "1.0.0"; }
PluginType GetTypeImpl() { return PLUGIN_TYPE_TOOLS; }

int InitializeImpl() {
    // 初始化资源，返回 1 成功，0 失败
    return 1;
}

char* HandleRequestImpl(const char* req) {
    auto request = json::parse(req);
    // 处理请求...

    json response;
    response["content"] = json::array();
    response["content"].push_back({{"type", "text"}, {"text", "result"}});
    response["isError"] = false;

    // 注意：返回的是 new 出来的内存，调用方负责 delete[]
    std::string result = response.dump();
    char* buffer = new char[result.length() + 1];
    strcpy(buffer, result.c_str());
    return buffer;
}

void ShutdownImpl() {
    // 清理资源
}

int GetToolCountImpl() { return sizeof(tools) / sizeof(tools[0]); }
const PluginTool* GetToolImpl(int index) {
    if (index < 0 || index >= GetToolCountImpl()) return nullptr;
    return &tools[index];
}

// 3. 填充接口表
static PluginAPI plugin = {
    GetNameImpl,       GetVersionImpl,    GetTypeImpl,
    InitializeImpl,    HandleRequestImpl, ShutdownImpl,
    GetToolCountImpl,  GetToolImpl,
    nullptr, nullptr, nullptr, nullptr
};

// 4. 导出工厂函数
extern "C" PLUGIN_API PluginAPI* CreatePlugin() {
    return &plugin;
}

extern "C" PLUGIN_API void DestroyPlugin(PluginAPI*) {
}

## 8. 立即加载与懒加载

### 8.1 RTLD_LAZY vs RTLD_NOW

```c
// 懒加载（Lazy Binding）
dlopen("libweather.so", RTLD_LAZY);

// 立即加载（Eager Binding）
dlopen("libweather.so", RTLD_NOW);
```

| 特性 | RTLD_LAZY（懒加载） | RTLD_NOW（立即加载） |
|------|---------------------|---------------------|
| 符号解析时机 | **首次调用时**解析 | **dlopen 时**全部解析 |
| dlopen 速度 | ✅ 快 | ❌ 慢（需解析所有符号） |
| 错误发现 | ❌ 运行时才报错 | ✅ 加载时就报错 |
| 内存使用 | 按需加载，更省内存 | 全部加载 |
| 适用场景 | 插件系统、不确定是否用到 | 启动时确保所有依赖就绪 |

### 8.2 本项目使用 RTLD_LAZY

```cpp
entry->handle = dlopen(stagingPath.c_str(), RTLD_LAZY);
```

**为什么选择 LAZY**：
1. 插件可能导出很多函数，但实际请求只调用一小部分
2. 如果某个依赖库暂时不可用，LAZY 模式下仍能加载成功
3. 加载速度更快，减少热替换时的卡顿

### 8.3 错误处理差异

```cpp
// RTLD_LAZY 的错误时机
handle = dlopen("lib.so", RTLD_LAZY);  // 成功！
func = dlsym(handle, "SomeFunc");       // 成功！拿到指针
func();                                  // ❌ 这里才报错：找不到依赖的 xxx 符号

// RTLD_NOW 的错误时机
handle = dlopen("lib.so", RTLD_NOW);   // ❌ 这里就报错：找不到 xxx 符号
// 或者成功，后续调用不会有符号解析错误
```

### 8.4 其他 dlopen 标志

```c
// 全局符号（默认）：符号对后续 dlopen 的库可见
dlopen("lib.so", RTLD_LAZY | RTLD_GLOBAL);

// 局部符号（推荐）：符号隔离，避免冲突
dlopen("lib.so", RTLD_LAZY | RTLD_LOCAL);  // 默认就是 LOCAL

// 不加入 dlclose 卸载列表：库永驻内存
dlopen("lib.so", RTLD_LAZY | RTLD_NODELETE);

// 深度绑定：解析所有依赖库的符号
dlopen("lib.so", RTLD_LAZY | RTLD_DEEPBIND);
```

谁分配 → 谁释放

## 10. 热插拔（Hot-Reload）机制

### 10.1 核心流程

```
启动
 ├─ LoadPlugins() 首次加载 plugins/ 下所有 .so
 ├─ StartWatching() 启动后台监控线程（每 5 秒扫描一次）
 └─ Server::Connect() 进入请求处理主循环

后台监控线程：
  WatchLoop()
    └─ 每 5 秒 → ScanForChanges()
         ├─ Phase 1: 读锁下收集差异（toAdd / toUpdate / toDelete）
         ├─ Phase 2: 无锁下创建新实例（dlopen + Initialize）
         └─ Phase 3: 写锁下原子替换（指针交换）
```

### 10.2 三阶段扫描详解

```
Phase 1: 分析（shared_lock）
┌────────────────────────────────────────────┐
│ 磁盘文件               vs    已加载插件     │
│────────────────────         ──────────────  │
│                                          │
│ 磁盘有、列表没有     ────→ toAdd          │
│ 两边都有、指纹不同   ────→ toUpdate       │
│ 磁盘没有、列表有     ────→ toDelete       │
│ 完全一致             ────→ 跳过           │
└────────────────────────────────────────────┘

Phase 2: 加载（无锁）
┌────────────────────────────────────────────┐
│ 对 toAdd 和 toUpdate 中的每个路径：         │
│   CopyToStaging → dlopen → CreatePlugin    │
│   → Initialize → 生成新 PluginEntry        │
│                                            │
│ 最耗时阶段，完全不持有任何锁                 │
└────────────────────────────────────────────┘

Phase 3: 替换（unique_lock）
┌────────────────────────────────────────────┐
│ 只做指针操作：                               │
│   m_plugins[i] = newEntry                  │
│   oldEntries.push_back(oldEntry)           │
│                                            │
│ 写锁内只做微秒级的指针交换                    │
└────────────────────────────────────────────┘

Phase 4: 收尾（无锁）
┌────────────────────────────────────────────┐
│ oldEntries.clear() → 释放旧插件             │
│ 清理 m_failedPlugins 中已删除的记录          │
│ 调用 OnPluginsChanged → 发送 list_changed   │
└────────────────────────────────────────────┘
```

### 10.3 请求线程的快照机制

```cpp
// 请求线程通过快照访问插件，无锁执行
auto plugins = loader->GetPluginsSnapshot();  // 读锁（微秒级）
// 锁已释放
for (const auto& plugin : plugins) {
    plugin->instance->HandleRequest(...);  // 无锁执行
}
// plugins 离开作用域，shared_ptr 引用计数减一
```

**关键保证**：
- 即使 watcher 线程在 HandleRequest 期间替换了插件，旧插件通过 `shared_ptr` 引用计数保持存活
- 只有最后一个快照释放时，旧插件才会真正析构

### 10.4 失败重试策略

```
插件加载失败
    │
    ├─ 原因是"源文件被修改"（kSourceChangedDuringCopy）
    │   → 不记入失败缓存，下轮直接重试
    │
    └─ 原因是插件本身问题（kLoadFailed）
        → 记录 mtime + size 到 m_failedPlugins
        → 后续扫描如果指纹未变，跳过
        → 只有文件被更新（新版本）才会重新尝试
```

## 11. 并发安全与线程模型

### 11.1 三线程模型

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  请求线程     │     │  Watcher线程  │     │  Writer线程   │
│  (主线程)     │     │  (后台扫描)   │     │  (通知发送)   │
├──────────────┤     ├──────────────┤     ├──────────────┤
│              │     │              │     │              │
│ GetPlugins   │     │ ScanFor      │     │ 等待通知队列  │
│ Snapshot()   │     │ Changes()    │     │              │
│  读锁(微秒)   │     │  读锁(分析)   │     │ transport    │
│              │     │  无锁(加载)   │     │ ->Write()    │
│ HandleReq()  │     │  写锁(切换)   │     │              │
│  无锁执行     │     │              │     │              │
└──────────────┘     └──────────────┘     └──────────────┘
```

### 11.2 并发控制机制

| 共享数据 | 保护方式 | 说明 |
|---------|---------|------|
| `m_plugins` | `shared_mutex` | 请求线程读锁、watcher 写锁 |
| `notification_queue` | `mutex` + `condition_variable` | Writer 线程消费 |
| `m_watching` | `atomic<bool>` | 控制 watcher 退出 |
| `isStopping_` | `atomic<bool>` | 控制主循环退出 |

### 11.3 PluginEntry 的 RAII 析构

```cpp
~PluginEntry() {
    if (instance) {
        instance->Shutdown();              // 1. 插件业务清理
        delete instance->notifications;    // 2. 释放宿主注入的通知系统
        instance->notifications = nullptr;
        destroyFunc(instance);             // 3. 调用 DestroyPlugin
        instance = nullptr;
    }
    if (handle) {
        dlclose(handle);                   // 4. 关闭动态库
        handle = nullptr;
    }
    if (!stagingPath.empty() && stagingPath != path) {
        std::filesystem::remove(stagingPath);  // 5. 删除 staging 副本
    }
}
```

## 12. 常见面试问题与回答要点

### Q1：为什么用 C 接口而不是 C++ 虚函数？

**答**：C ABI 在所有编译器（GCC/Clang/MSVC）间保持稳定，而 C++ 的虚表布局、名称修饰（name mangling）在不同编译器甚至不同版本间可能不兼容。用 `extern "C"` 导出函数 + 函数指针表模拟虚表，是插件系统的通用做法。

### Q2：`RTLD_LAZY` 和 `RTLD_NOW` 的区别？

**答**：`RTLD_LAZY` 在首次调用函数时才解析符号，加载快但运行时可能报错；`RTLD_NOW` 在 `dlopen` 时就解析所有符号，加载慢但能尽早发现错误。插件系统常用 `RTLD_LAZY` 以加快加载速度。

### Q3：为什么需要 staging 机制？

**答**：Linux 的 `dlopen` 对同一路径有模块缓存。如果直接 `dlopen` 原始路径，更新文件后再次 `dlopen` 同一路径，操作系统返回已缓存的旧模块。通过复制到带时间戳的唯一路径，确保每次加载都是新模块。

### Q4：shared_ptr 如何保证并发安全？

**答**：请求线程通过 `GetPluginsSnapshot()` 获取 `vector<shared_ptr<PluginEntry>>` 的拷贝。即使 watcher 线程同时替换了插件，请求线程持有的 `shared_ptr` 引用保持旧插件存活。只有最后一个引用释放时，旧插件才通过 RAII 析构。读写锁保护 `m_plugins` 列表，`shared_ptr` 引用计数保证对象生命周期。

### Q5：动态库和静态库的区别？什么时候用哪个？

**答**：
- **动态库**：运行时加载，多个进程共享同一份代码，适合插件系统、需要热更新的场景
- **静态库**：编译时链接，代码复制进可执行文件，无运行时依赖，适合发布独立工具

### Q6：`__attribute__((visibility("default")))` 的作用？

**答**：配合 `-fvisibility=hidden` 编译选项，默认隐藏所有符号，只导出标记的符号。好处：减少符号表大小、避免符号冲突、加快加载速度、缩小 .so 体积。

### Q7：热插拔的三阶段扫描为什么这样设计？

**答**：
1. **读锁阶段**：分析差异，持有时间极短，不影响请求处理
2. **无锁阶段**：最耗时的文件 I/O 和 dlopen 操作，在锁外执行
3. **写锁阶段**：只做指针交换，微秒级完成

这种设计最大限度减少了锁持有时间，请求线程几乎不会被阻塞。

### Q8：`extern "C"` 在这里的作用？

**答**：防止 C++ 的名称修饰（name mangling）。C++ 编译器会将函数名编码为包含参数类型等信息的复杂名称（如 `_Z13CreatePluginv`），而 `extern "C"` 保持函数名为原始的 `CreatePlugin`，使 `dlsym` 能正确找到。

### Q9：插件内存管理中的坑？

**答**：
- `HandleRequest` 返回 `new char[]`，调用方必须 `delete[]`，否则内存泄漏
- `CreatePlugin` 返回静态对象指针时，`DestroyPlugin` 不应 `delete`
- 如果 `CreatePlugin` 返回 `new` 出的对象，`DestroyPlugin` 必须对应 `delete`
- **谁分配谁释放**原则：主程序 `new` 的 `NotificationSystem` 由主程序在析构时 `delete`

### Q10：`-fPIC` 为什么是动态库必须的？

**答**：位置无关代码（Position Independent Code）。动态库加载到内存的地址在编译时不确定，需要通过 GOT（全局偏移表）和 PLT（过程链接表）间接寻址。`-fPIC` 生成的代码可以被加载到任意地址运行，不需要修改代码段，使得多个进程可以共享同一份物理页面。
