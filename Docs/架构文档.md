## 各层职责

### 1. Transport Layer（传输层）

**职责：** 将底层通信协议（stdin/stdout、HTTP、SSE）抽象为统一的 `ITransport` 接口，屏蔽协议差异。

**接口定义 (`ITransport`)：**

| 方法 | 说明 |
|------|------|
| `Start()` / `Stop()` | 生命周期管理 |
| `Read()` / `ReadAsync()` | 从客户端读取一条 JSON-RPC 消息 |
| `Write()` / `WriteAsync()` | 向客户端写回一条响应或通知 |

**实现：**

| 实现类 | 协议 | 场景 |
|--------|------|------|
| `Stdio` | stdin/stdout | CLI 调用，与 Claude Desktop 等集成 |
| `SSE` | HTTP + Server-Sent Events | 浏览器/远程客户端 |
| `HttpStream` | HTTP 长连接流 | 早期 HTTP 传输 |
| `StreamableTransport` | HTTP POST/GET/DELETE | MCP 2025-03-26 规范，支持 JSON/SSE 双模式响应 |

### 2. Server Layer（服务层）

**职责：** MCP 协议核心，实现 JSON-RPC 路由、请求分发、异步通知队列。

**核心组件：**

- **`functionMap`：** 方法名 → 处理函数的映射表，实现 JSON-RPC 方法路由
- **Reader Loop：** 循环调用 `transport_->Read()`，解析请求并分发到对应处理函数
- **Writer Loop：** 独立线程，从通知队列取出消息异步发送，避免阻塞主流程
- **`OverrideCallback()`：** 允许外部（main 函数）覆盖内置处理逻辑，实现 Server 与 PluginLoader 的解耦连接

**数据流：**

```
Client → Transport.Read() → Server.HandleRequest() → functionMap[method]() → Transport.Write()
                                                                          ↕
                                                            notification_queue_ → WriterLoop → Transport.Write()
```

### 3. PluginLoader Layer（插件加载层）

**职责：** 动态加载、卸载、热监听插件 `.so` 文件，维护插件实例的线程安全管理。

**核心机制：**

| 机制 | 说明 |
|------|------|
| `dlopen/dlsym` | 动态加载 `.so`，通过 `CreatePlugin` 导出符号获取 `PluginAPI*` |
| Staging Copy | 加载前将 `.so` 复制到临时目录，避免编译/替换时文件锁定问题 |
| 文件指纹监听 | 后台线程定期扫描目录，比对 `mtime + size`，检测插件变更 |
| `shared_mutex` | 读写锁保护插件列表，`GetPluginsSnapshot()` 返回线程安全快照 |
| 回调通知 | `OnPluginLoaded` / `OnPluginsChanged` 回调，通知 Server 层插件变化 |

### 4. Plugin Layer（插件层）

**职责：** 独立的功能单元，以 `.so` 动态库形式存在，通过 C ABI 对外暴露能力。

**接口定义 (`PluginAPI`)：**

| 函数指针 | 说明 |
|----------|------|
| `GetName/GetVersion/GetType` | 插件元信息 |
| `Initialize / Shutdown` | 生命周期 |
| `HandleRequest` | 处理 JSON-RPC 请求，返回 `char*`（调用方负责释放） |
| `GetToolCount/GetTool` | 暴露 Tools（类型为 `PLUGIN_TYPE_TOOLS`） |
| `GetPromptCount/GetPrompt` | 暴露 Prompts（类型为 `PLUGIN_TYPE_PROMPTS`） |
| `GetResourceCount/GetResource` | 暴露 Resources（类型为 `PLUGIN_TYPE_RESOURCES`） |
| `notifications->SendToClient` | 插件主动向客户端推送通知 |

**插件类型：** 每个插件声明一种 `PluginType`（TOOLS / PROMPTS / RESOURCES），Server 按类型聚合。

## 层间通信

```
Transport ◄───ITransport 接口────► Server
                                      │
                                      │ OverrideCallback 闭包（注册路由处理函数）
                                      │ GetPluginsSnapshot（获取插件快照）
                                      ▼
                                 PluginLoader
                                      │
                                      │ dlopen / dlsym / PluginAPI C ABI
                                      ▼
                                   Plugin (.so)
                                      │
                                      │ NotificationSystem::SendToClient 回调
                                      ▼
                                 Server → Transport → Client
```

- **Transport ↔ Server：** 通过 `ITransport` 虚接口，Server 不关心具体协议实现
- **Server ↔ PluginLoader：** 通过 `main()` 中的 `OverrideCallback` 闭包桥接，Server 本身不依赖 PluginLoader
- **PluginLoader ↔ Plugin：** 通过 C ABI（`extern "C"` 导出函数），ABI 稳定，跨编译器兼容
- **Plugin → Client：** 通过 `NotificationSystem::SendToClient` 回调，经由 Server 通知队列异步发送

## 隔离设计

| 隔离维度 | 实现方式 |
|----------|----------|
| **协议隔离** | `ITransport` 接口抽象，Transport 实现可自由替换，Server 层零改动 |
| **插件进程隔离** | 插件以 `.so` 形式加载在主进程，通过 C ABI 边界隔离（非进程级） |
| **插件数据隔离** | 插件仅通过 `PluginAPI` 函数指针与宿主通信，无法直接访问 Server 内部状态 |
| **热加载安全** | Staging Copy 机制确保 `dlopen` 的文件不会被外部覆盖导致崩溃 |
| **线程安全** | `shared_mutex` 保护插件列表读写；`mutex + condition_variable` 保护通知队列 |
| **生命周期隔离** | `PluginEntry` 析构时自动调用 `Shutdown()` → `DestroyPlugin()` → `dlclose()` |
| **Server-Loader 解耦** | Server 不包含任何 PluginLoader 头文件，通过 `OverrideCallback` 回调注入插件逻辑 |
