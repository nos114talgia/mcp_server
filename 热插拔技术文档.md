# MCP Server 插件热插拔技术文档

## 1. 功能概述

MCP Server 支持在运行过程中自动检测插件目录的变化，无需重启即可完成插件的新增、更新和删除。后台监控线程每 5 秒扫描一次插件目录，发现变化后自动加载新插件或替换旧插件，并通过 MCP 协议通知客户端刷新能力列表。

## 2. 整体流程

服务器启动到运行的完整流程如下：

```
启动
 │
 ├─ 1. 设置回调（OnPluginLoaded / OnPluginsChanged）
 ├─ 2. LoadPlugins() 首次加载所有插件
 ├─ 3. StartWatching() 启动后台监控线程
 ├─ 4. server->Connect() 进入请求处理主循环
 │       │
 │       ├─ 请求线程：GetPluginsSnapshot() 获取插件快照 → 调用插件
 │       └─ Watcher 线程：每 5 秒 ScanForChanges() 检测变化
 │
 └─ 收到 SIGINT → Connect() 退出 → StopWatching() → UnloadPlugins() → 退出
```

## 3. 涉及的文件与核心类

| 文件                | 职责                                                              |
| ------------------- | ----------------------------------------------------------------- |
| `PluginsLoader.h`   | 定义 `PluginEntry`（插件生命周期结构体）和 `PluginsLoader` 类     |
| `PluginsLoader.cpp` | 实现插件加载、staging 机制、后台监控、三阶段扫描                  |
| `PluginAPI.h`       | 定义插件接口（`CreatePlugin`/`DestroyPlugin`/`HandleRequest` 等） |
| `main.cpp`          | 配置回调、启动监控、请求处理中通过快照访问插件                    |
| `MCPBuilder.h`      | 构造 `list_changed` 通知消息                                      |
| `Server.h/cpp`      | 请求处理主循环、通知发送队列、`RequestStop()` 安全停止            |

## 4. 插件的加载过程

每个插件从磁盘文件到可服务状态，经过以下步骤：

```
原始文件 (plugins/libweather.so)
    │
    ▼
① 采集文件指纹（mtime + size）
    │
    ▼
② 复制到 staging 目录，文件名带纳秒时间戳
   (plugins/.staging/libweather_1719000001.so)
    │
    ▼
③ 校验源文件在复制期间未被修改（再次读取 mtime + size 对比）
    │
    ▼
④ dlopen() 加载 staging 副本
    │
    ▼
⑤ dlsym() 获取 CreatePlugin / DestroyPlugin 函数指针
    │
    ▼
⑥ 调用 CreatePlugin() 创建插件实例
    │
    ▼
⑦ 调用 instance->Initialize() 初始化插件
    │
    ▼
⑧ 触发 OnPluginLoaded 回调（设置通知系统等宿主侧初始化）
    │
    ▼
⑨ 返回 shared_ptr<PluginEntry>，由调用方加入活跃插件列表
```

**staging 机制**：不直接 dlopen 原始路径，而是复制到 `.staging/` 下带唯一时间戳的路径再加载。这是因为插件内部使用 `static PluginAPI plugin` 静态对象，如果对同一路径多次 dlopen，操作系统会复用已加载的模块而不是加载新文件。通过不同的 staging 路径，确保每次加载都是独立的模块实例。

**指纹校验**：在复制前后各读取一次源文件的 mtime 和 size。如果不一致，说明复制期间文件被构建系统覆盖了，本次加载的内容可能不完整，丢弃 staging 副本，等待下一轮扫描重试。

## 5. 后台监控与三阶段扫描

`StartWatching()` 启动一个后台线程，每隔指定间隔（默认 5 秒）执行一次 `ScanForChanges()`。

### 5.1 第一阶段：读锁下收集差异

持有 `shared_lock`，对比磁盘文件和已加载插件列表：

```
磁盘文件 currentFiles    vs    已加载插件 m_plugins
─────────────────────         ──────────────────
                                                    
找不到已加载插件的文件  ───────→  标记为 toDelete     
mtime 或 size 不一致   ───────→  标记为 toUpdate     
磁盘有但列表没有       ───────→  标记为 toAdd        
全都一致              ───────→  跳过                
```

此阶段还会检查失败缓存 `m_failedPlugins`：如果某个文件之前加载失败且 mtime/size 未变化，直接跳过，不重复尝试。

### 5.2 第二阶段：锁外创建新实例

释放所有锁，对 toUpdate 和 toAdd 中的每个路径执行第 4 节描述的完整加载流程。这是最耗时的阶段（文件 I/O、dlopen、Initialize），完全不持任何锁，不影响正在处理的请求。

加载结果有三种：

| 结果                       | 含义                 | 后续处理                         |
| -------------------------- | -------------------- | -------------------------------- |
| `kSuccess`                 | 加载成功             | 清除该路径的失败缓存             |
| `kLoadFailed`              | 插件本身有问题       | 记入失败缓存，文件不变则不再重试 |
| `kSourceChangedDuringCopy` | 复制期间源文件被修改 | 不记入缓存，下轮自动重试         |

### 5.3 第三阶段：写锁下原子切换

持有 `unique_lock`，对 `m_plugins` 执行实际修改：

- **更新**：将新实例与旧实例做指针交换，旧实例移入 `oldEntries` 待释放列表
- **删除**：二次确认文件确实不存在后，从列表移除并移入 `oldEntries`
- **新增**：将新实例加入列表

写锁内只做指针操作，不执行任何耗时操作。

### 5.4 锁外收尾

```
oldEntries.clear()
    │
    │  旧插件的 shared_ptr 在这里释放。
    │  如果还有请求快照持有引用，析构会推迟到快照释放时。
    │
    ▼
清理 m_failedPlugins 中文件已不存在的记录
    │
    ▼
调用 OnPluginsChanged 回调，按类型发送 list_changed 通知
```

## 6. 请求处理中的快照访问

请求线程通过 `GetPluginsSnapshot()` 获取当前插件列表的副本：

```cpp
auto plugins = loader->GetPluginsSnapshot();  // 读锁，拷贝 vector<shared_ptr>，释放锁
for (const auto& plugin : plugins) {
    plugin->instance->HandleRequest(...);     // 无锁执行，可以是耗时操作
}
// plugins 离开作用域，shared_ptr 引用计数减一
```

这个机制的关键点：
- 读锁只在拷贝 vector 时持有（微秒级），立即释放
- `HandleRequest()` 在完全无锁状态下执行
- 快照持有 `shared_ptr` 引用，即使 watcher 线程同时替换了插件，旧插件在快照释放前不会被卸载

## 7. 插件卸载过程

`PluginEntry` 的析构函数定义了完整的卸载链：

```
~PluginEntry()
    │
    ├─ instance->Shutdown()         调用插件的关闭方法
    ├─ delete instance->notifications   释放宿主分配的通知系统
    ├─ destroyFunc(instance)        调用 DestroyPlugin()
    ├─ dlclose(handle)              卸载动态库
    └─ remove(stagingPath)          删除 staging 副本文件
```

析构时机由 `shared_ptr` 的引用计数决定：
- watcher 线程替换插件时，旧 `shared_ptr` 移出列表
- 如果没有请求快照持有引用，立即析构
- 如果有请求快照持有引用，推迟到最后一个快照释放时析构

## 8. 线程模型与并发控制

系统中有三个线程：

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   主线程      │     │  Watcher线程  │     │  Writer线程   │
│  (请求处理)   │     │ (目录扫描)    │      │ (通知发送)    │
├──────────────┤     ├──────────────┤     ├──────────────┤
│              │     │              │     │              │
│ GetPlugins   │     │ ScanFor      │     │ WriterLoop   │
│ Snapshot()   │     │ Changes()    │     │              │
│   读锁(短暂)   │     │  读锁(分析)   │     │ 等待通知队列  │
│              │     │  无锁(加载)   │     │              │
│ HandleReq()  │     │  写锁(切换)   │     │ transport    │
│   (无锁)      │     │              │     │ ->Write()    │
│              │     │  回调通知     │     │              │
└──────────────┘     └──────────────┘     └──────────────┘
```

并发控制机制：

| 共享数据             | 保护方式                       | 说明                           |
| -------------------- | ------------------------------ | ------------------------------ |
| `m_plugins`          | `shared_mutex`                 | 请求线程读锁、watcher 线程写锁 |
| `notification_queue` | `mutex` + `condition_variable` | Writer 线程消费                |
| `m_watching`         | `atomic<bool>`                 | 控制 watcher 线程退出          |
| `isStopping_`        | `atomic<bool>`                 | 控制请求主循环退出             |

## 9. 客户端通知

插件变化后，服务器通过 MCP 协议通知客户端。

**能力声明**（InitializeCmd 响应）：

```json
{
  "capabilities": {
    "tools": { "listChanged": true },
    "prompts": { "listChanged": true },
    "resources": { "listChanged": true, "subscribe": true }
  }
}
```

**通知发送**：`ScanForChanges` 在第三阶段记录每种插件类型是否发生了变化。替换场景下会同时标记旧插件类型和新插件类型（因为一个插件更新后类型可能改变）。最终只发送实际变化类型对应的通知：

```
toolsChanged == true    →  发送 notifications/tools/list_changed
promptsChanged == true  →  发送 notifications/prompts/list_changed
resourcesChanged == true → 发送 notifications/resources/list_changed
```

## 10. 失败重试策略

为避免损坏的插件每轮扫描都被重复尝试加载（产生 I/O 和日志噪音），系统维护了一个失败指纹缓存 `m_failedPlugins`：

```
首次加载失败
    │
    ├─ 记录当前文件的 mtime + size 到 m_failedPlugins
    │
    ▼
后续扫描
    │
    ├─ 文件 mtime/size 与缓存一致 → 跳过，不重试
    ├─ 文件 mtime/size 发生变化   → 清除缓存，重新尝试加载
    └─ 文件已被删除              → 清除缓存记录
```

特殊情况：如果加载失败的原因是"复制期间源文件被修改"（`kSourceChangedDuringCopy`），不记入失败缓存，下一轮直接重试。因为这不是插件本身的问题，只是时机问题。

## 11. 安全退出

```
用户按下 Ctrl+C
    │
    ▼
stop_handler()（信号处理函数）
    ├─ 设置 g_stopRequested = 1     （volatile sig_atomic_t）
    └─ 调用 server->RequestStop()   （只设置 atomic<bool> 标志）
    │
    ▼
主循环检测到 isStopping_ 为 true，Connect() 返回
    │
    ▼
main() 中顺序执行清理：
    ├─ loader->StopWatching()    等待 watcher 线程结束
    ├─ loader->UnloadPlugins()   释放所有插件
    └─ server->Stop()            关闭 transport，等待 writer 线程结束
```

信号处理函数中只执行信号安全操作（设置原子标志），所有实际清理工作在 `Connect()` 返回后由主线程执行。

## 12. 目录结构

```
plugins/
└── libweather.so             

```

- `.staging/` 目录由 `LoadPlugins()` 或 `StartWatching()` 首次调用时自动创建
- 扫描时通过 `IsStagingPath()` 跳过该目录下的文件，避免把副本当作新插件
- 各插件卸载时自动清理自己的 staging 文件
- 进程异常退出可能残留文件，不影响下次启动
